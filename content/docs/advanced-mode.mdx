---
title: Advanced Mode
description: Full control over the engine — runtime, providers, budgets, prediction policy, Deep-Run, and integration.
---

Advanced Mode exposes every knob the engine actually consults. Most
users do not need it: the seven shipped [policy bundles](/policy-bundles)
already cover the vast majority of working configurations, and Simple
Mode picks one for you.

Reach for Advanced Mode when:

- you need to wire Vaner into a custom backend (a non-default runtime,
  a self-hosted gateway, a corporate proxy)
- you want to tune drafting aggressiveness, exploration ratio, or
  persistence strength beyond what the bundle defaults give you
- you are building Vaner into a product and need deterministic config
  rather than auto-selection

For raw config syntax (file format, env-var overlays, CLI shortcuts),
see [Configuration](/configuration). The sections below mirror the
seven groupings the engine uses internally and the desktop apps
expose as Preferences tabs.

## 1. Runtime and providers

Where models actually run. If you only have one local runtime
installed, Vaner detects it and you can leave this alone.

| Field path | Purpose |
|---|---|
| `[backend.local_runtime]` | The local LLM runtime. One of `ollama`, `llama.cpp`, `lmstudio`, `vllm`, `mlx`, or `auto` for first-detected. |
| `[backend.base_url]` | OpenAI-compatible endpoint URL the engine talks to (defaults to the local runtime's). |
| `[backend.model]` | The model id served by `base_url`. |
| `[backend.api_key_env]` | Name of the env var holding the API key, if the endpoint requires one. |
| `[backend.cloud_provider]` | Optional cloud provider id (e.g. `openai`, `anthropic`) for hybrid bundles. |
| `[backend.routing]` | `local`, `cloud`, or `auto` — which side of the runtime split a given task lands on. |
| `[backend.model_family]`, `[backend.model_size]` | Hints used by the bundle resolver when `model` is left unset. |

**Tuning guidance.** Start by leaving `local_runtime = "auto"` and
letting the hardware probe pick. Override `routing = "local"` if you
want to disable cloud calls without touching the cloud provider id
(useful when a hybrid bundle is *otherwise* the right shape but you're
on the train).

## 2. Budgets

Caps that prevent runaway cost or cycle time.

| Field path | Purpose |
|---|---|
| `[compute.max_cycle_seconds]` | Wall-clock cap for a single ponder cycle. Defaults to 300 s. |
| `[compute.max_session_minutes]` | Total time the daemon can spend in a session. |
| `[compute.max_concurrent_jobs]` | Parallel cycles allowed at once. Bigger boxes can raise this; light hardware should leave it at the default. |
| `[budgets.daily_cost_cap]` | USD cap on cloud spend per day. Hybrid bundles set sensible defaults; cost-sensitive users should drop this further. |
| `[budgets.tokens.foreground]`, `[budgets.tokens.background]`, `[budgets.tokens.deep_run]` | Per-mode token caps. Foreground = user-facing answers; background = idle pondering; Deep-Run = explicit overnight sessions. |

**Tuning guidance.** Cycle time and concurrent jobs are the two knobs
that change how *much* Vaner does. Cost / token caps change *what it
costs you*. Set the dollar cap first, then dial the cycle/job knobs
to match the hardware tier — the [hardware tiers](/hardware-tiers)
page lists sensible starting values per tier.

## 3. Reasoning and drafting

How the drafter decides whether an answer is ready, and how
aggressively it pushes through partial evidence.

| Field path | Purpose |
|---|---|
| `[drafting.reasoning_mode]` | `fast`, `balanced`, or `deep` — how many reasoning passes the drafter takes. |
| `[drafting.aggressiveness]` | `0.7` (cautious) → `1.3` (eager). Multiplier on the drafter's evidence threshold; higher = ship the draft sooner. |
| `[drafting.evidence_threshold]` | Raw threshold; aggressiveness multiplies into this. |
| `[drafting.contradiction_behavior]` | `flag`, `defer`, or `block` — what the drafter does when retrieved evidence contradicts itself. |

**Tuning guidance.** Bundle defaults are calibrated for the bundle's
target user; only override if you have a measurable answer-quality
gap. Bumping `aggressiveness` is the cheapest way to make Vaner
*feel* faster, at the cost of more incomplete drafts.

## 4. Prediction policy

How Vaner allocates background pondering across short-horizon
("what's likely next?") and long-horizon ("what will matter
tomorrow?") work.

| Field path | Purpose |
|---|---|
| `[prediction.horizon_bias]` | Weight distribution over four buckets — `likely_next`, `long_horizon`, `finish_partials`, `balanced`. The engine normalises before applying. |
| `[prediction.exploration_ratio]` | `-0.10` (exploit-leaning) → `+0.15` (explore-leaning). Additive bias on top of the base exploration weights. |
| `[prediction.persistence_strength]` | `0.6` (decay fast) → `1.5` (decay slow). Multiplier on memory retention — how long Vaner keeps a prediction "warm". |
| `[prediction.goal_weighting]` | `0.8` → `1.4`. Multiplier on goal-aligned scoring; raise it when you have explicit goals you want Vaner to bias toward. |

**Tuning guidance.** These four knobs interact strongly. The bundle
calibration in
[`catalog.py`](https://github.com/Borgels/vaner/blob/main/src/vaner/setup/catalog.py)
is your reference for sane combinations — start from a bundle and
nudge one axis at a time.

## 5. Deep-Run

The overnight / long-horizon mode. See [Deep-Run](/deep-run) for the
full lifecycle.

| Field path | Purpose |
|---|---|
| `[deep_run.preset]` | `conservative`, `balanced`, or `aggressive`. Each bundle sets a default preset; Deep-Run sessions can override. |
| `[deep_run.max_duration_minutes]` | Hard cap on a single Deep-Run session. |
| `[deep_run.locality]` | `local_only`, `hybrid`, `cloud_preferred` — overrides the bundle's `local_cloud_posture` for this session only. |
| `[deep_run.cost_cap_usd]` | Per-session cloud spend cap. |
| `[deep_run.focus]` | Free-text focus hint passed to the planner. |
| `[deep_run.prefer]` | Optional list of skills, scenarios, or work-styles to bias toward. |

**Tuning guidance.** The sensible default is *don't override*. Deep-Run
defaults are derived from the active bundle's `deep_run_profile`
(see [Policy bundles](/policy-bundles)). Override per-session via the
"Start Deep-Run" dialog or `GET /deep-run/defaults` for programmatic
consumers.

## 6. Integration

How Vaner interacts with your IDE / chat client.

| Field path | Purpose |
|---|---|
| `[integration.guidance_injection]` | `off`, `passive`, `active` — whether Vaner injects guidance text into the model's system prompt. |
| `[integration.context_injection.mode]` | `none`, `digest_only`, `adopted_package_only`, `top_match_auto_include`, `policy_hybrid`, `client_controlled`. |
| `[integration.mcp_apps_enabled]` | Toggle the MCP Apps UI surface (introduced in 0.8.5). |
| `[integration.handoff_method]` | How Vaner ships prepared context to the model — inline, attachment, or resource link. |

**Tuning guidance.** `policy_hybrid` is the right default for almost
all bundles. Drop to `digest_only` if your client's context window is
tight; use `client_controlled` when you want the IDE to drive
inclusion explicitly. See
[Integrations / Context-injection layer](/integrations/integration-layer)
for the full mode reference.

## Where to set these

- `.vaner/config.toml` — persistent file config.
- `vaner config set <key> <value>` — CLI shortcut, hot-reloads the
  daemon if one is running.
- `vaner setup advanced` — opens the file in `$EDITOR`.
- Cockpit Preferences → Engine (Advanced tab) — interactive editor.

When a knob is bundle-managed (i.e. the active bundle has set it for
you), the cockpit shows the field as read-only with a *"Switch to
Advanced"* affordance. Switching marks your overrides as deviations
in `PolicyConfig.bundle_overrides`; the rest of the bundle still
applies.
